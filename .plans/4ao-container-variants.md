# Base container image + overlay system (agent-in-docker-4ao)

## Problem
Currently one container image for all agents. Different workflows need different tools: a Python data agent needs pandas/numpy, a Rust agent needs cargo, a web agent needs more browser tools. Adding everything to one image makes it bloated.

## Design: Base Image + Overlay Containerfiles

### Base image (`Containerfile.base`)
Contains everything shared:
- node:22-alpine + gcompat
- Claude Code CLI
- Python 3, Git, bash, curl
- Chromium (Playwright)
- beads (bd) + dolt
- entrypoint.sh

Build: `podman build -f Containerfile.base -t agent-base .`

### Variant Containerfiles (`variants/`)
Each extends the base:

```
variants/
  python-data.Containerfile    # pandas, numpy, jupyter
  rust-dev.Containerfile       # rustup, cargo
  web-dev.Containerfile        # node_modules, playwright browsers
  minimal.Containerfile        # just claude + git, smallest possible
```

Example `variants/python-data.Containerfile`:
```dockerfile
FROM agent-base
RUN pip3 install pandas numpy matplotlib jupyter
```

Example `variants/rust-dev.Containerfile`:
```dockerfile
FROM agent-base
RUN curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y
ENV PATH="/root/.cargo/bin:${PATH}"
```

### CLI integration
The `--role` flag maps to a variant image:
- `--role code-agent` → `agent-in-docker` (base, default)
- `--role python-agent` → `agent-python-data`
- `--role rust-agent` → `agent-rust-dev`

Or add an explicit `--image` flag:
```bash
agent run . "Analyze the data" --image agent-python-data
```

### Role → image mapping
In `roles/*.yml`, add an `image` field:
```yaml
name: python-agent
image: agent-python-data
capabilities:
  file_read: true
  # ...
```

The CLI reads the role file, finds the image name, and uses it for podman run.

### Custom variants
Users create their own by writing a Containerfile that extends `agent-base`:
```dockerfile
FROM agent-base
RUN apk add --no-cache my-custom-tool
COPY my-mcp-config.json /opt/extra-mcp.json
```

### MCP tool extension
Variants can add MCP servers by including additional configs. The entrypoint merges MCP configs from:
1. `/tmp/mcp-config.json` (generated by entrypoint -- the orchestrator bridge)
2. `/opt/extra-mcp/*.json` (variant-provided)

## Implementation

### Step 1: Split Containerfile into base
- Rename current `Containerfile` → `Containerfile.base`
- Create `Containerfile` that just `FROM agent-base` (no-op overlay for default)
- Update CLI to build `agent-base` first, then variant

### Step 2: Create variant directory
- `variants/python-data.Containerfile`
- `variants/rust-dev.Containerfile`
- `variants/minimal.Containerfile`

### Step 3: CLI build logic
```rust
// In container.rs
fn ensure_base_image(cfg) -> Result<()> {
    if !image_exists("agent-base")? {
        build("Containerfile.base", "agent-base");
    }
}

fn ensure_variant_image(variant: &str) -> Result<()> {
    ensure_base_image()?;
    let containerfile = format!("variants/{}.Containerfile", variant);
    if Path::new(&containerfile).exists() {
        build(&containerfile, &format!("agent-{}", variant));
    }
}
```

### Step 4: Role → image mapping
- Add `image` field to Role struct in `permissions.rs`
- CLI reads role file, gets image name
- Falls back to "agent-in-docker" if no image specified

### Files to modify
- `Containerfile` → `Containerfile.base` + `Containerfile`
- `variants/` -- new directory with variant Containerfiles
- `orchestrator/crates/cli/src/container.rs` -- build logic for base + variants
- `orchestrator/crates/core/src/permissions.rs` -- add `image` to Role
- `roles/*.yml` -- add image field
